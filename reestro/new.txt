В reestro_parser.py добавлена защита от обрывов:

Изменение	Зачем
5 повторов при таймауте / обрыве соединения
Пауза 3, 6, 9… сек между попытками
--timeout 120 (было 60)
Больше времени на медленный канал
--pause 2 (было 0.4)
Меньше нагрузки на API и файрвол
Обрыв сети → пропуск без записи
Объект не попадает в report, при следующем запуске повторится
--save-every 5
report сохраняется каждые 5 объектов (не теряется при падении)
Прокси
В config.json: "proxy": "http://..." или переменные HTTP_PROXY / HTTPS_PROXY
Команда для заказчика (нестабильная сеть)
python reestro_parser.py -i TZ\Запрос.xlsx -o output -c config.json --pause 3 --timeout 120 --retries 5 --save-every 1
Параметр	Значение
--pause 3
3 сек между запросами
--timeout 120
ждать ответ до 2 минут
--retries 5
до 5 попыток при обрыве
--save-every 1
сохранять report после каждого объекта
Партиями (если сеть очень слабая):

python reestro_parser.py -i TZ\Запрос.xlsx -o output -c config.json --limit 20 --pause 3 --timeout 120 --save-every 1
Повторять ту же команду — готовые 85 объектов пропустятся.

Как работает при обрыве
Запрос к API
    ↓
Обрыв / таймаут → повтор (до 5 раз)
    ↓
Все попытки неудачны → «сеть: … — пропуск»
    ↓
Объект НЕ в report, НЕ в PDF → при перезапуске снова в очереди
Раньше скрипт падал на первом обрыве. Теперь идёт дальше.

Что проверить у IT заказчика
Исходящий HTTPS на api.kontur.ru:443 разрешён.
Если есть корпоративный прокси — прописать в config.json:
{
  "apiKey": "...",
  "orgId": "...",
  "baseUrl": "https://api.kontur.ru",
  "proxy": "http://proxy.company.ru:8080"
}
report.xlsx закрыт в Excel во время работы парсера.
Проверка с той же машины:
python check_balance.py
